API design principles

Design principles of AutoScript

The following design principles provide a clear set of guidelines to ensure the API's usability, stability, and consistency:

  1. Easy to use:

    2. The API should be user-friendly while offering powerful functionality.

  2. Minimizing breaking changes:

    3. Changes that could break existing scripts are minimized. Experimental features are marked, indicating a higher likelihood of breaking changes (e.g. autofunctions).

  3. SI units are preferred:

    4. Standard International (SI) units are preferred for consistency and clarity.

  4. Consistency:

    5. Similar functionalities should behave in a consistent manner for better user understanding.

  5. Explicit errors instead of silent ones:

    6. When functionality doesn't meet user expectations, raise explicit errors rather than failing silently. For example, if a function is called with an invalid combination of arguments, an error should be raised instead of presuming what the user probably intended. This guides the user to use the API correctly instead of guessing.

    One exception to this rule is when the user tries to set a state that is already set in the system. In this case, the call is executed without an error, but the system state remains unchanged.

  6. Same API on every type of microscope:

    7. Scripts using the API should be portable across different systems with the same capabilities.

  7. Minimizing internal state:

    8. AutoScript server is designed to have as little internal state as possible since the microscope server itself should hold state information. In cases where this is not possible, the state is not shared between different clients/scripts, i.e. each client has its own internal state. This reduces the complexity of the entire script ecosystem. In most cases, the only places where a changing internal state needs to be taken into account are the microscope server and the user script itself.

Reoccurring patterns

The AutoScript API is designed to be consistent. Concepts with similar functionality are modeled in the same way and their behavior conveys further information about the microscope's internal state. The following are some of the recurring patterns in the API:

  1. Value and available values pair

    2. Properties representing a state have a value property and a method returning a collection of available values. The value property represents the current state and the available values method returns a collection of currently accessible states. An empty collection signifies that no values are accessible at the current microscope state and an exception is thrown when the values are inaccessible due to missing hardware.

  2. Value and limits pair

    3. Functions that retrieve or adjust continuous values within allowable ranges are usually represented by a pair of properties: value and limits. The limits property returns the Limits structure that defines the upper and lower value thresholds.

  3. Settings structures

    4. Functions that take numerous optional input parameters are designed to accept a single "settings" parameter, as an instance of a data class with optional items. This approach allows the settings structures to be easily expanded with new items without having to change function signatures. The naming convention for the settings structure often mirrors the name of the function for which it is intended. For example, RunStemAutoFocusSettings is a settings structure intended for the auto_functions.run_stem_auto_focus() function. However, there are exceptions to this convention, where the settings structure serves a broader purpose, for instance is utilized for multiple functions.

Synchronicity

By default, functions in the API are designed to be synchronous, meaning they return whenever all effects of the action on the microscope hardware and software have subsided. There are some exceptions to this rule, whenever an asynchronous version of the function is available, it is usually named with the word "Start" to denote the asynchronicity. For example microscope.specimen.stage.start_jogging() starts the jogging process and returns control immediately back to the script (without waiting for the jogging to finish).

Breaking change strategy

The following breaking change strategy provides a structured approach to manage intentional changes that could impact existing users:

  1. Documentation:

    2. All intentional breaking changes are documented in the Release notes.

  2. Smaller breaking changes:

    3. Smaller changes or fixes are usually documented without additional actions.

  3. Larger breaking changes:
    • Old functionality is marked as obsolete.
    • A recommended alternative is provided.
    • At least four full version releases have to pass before the old feature is removed.
    • Obsoleted parts of the API are removed.

Changes considered as breaking

  1. Removal or renaming of any part of the API. This includes groups, functions, enums, structures, etc.
  2. Changing the order of function arguments.
  3. Changing the order of arguments in structure initializations.
  4. Removing or changing part of metadata.
  5. Removing support for file formats. Specific image encodings and bit-depth combinations, emd file versions, etc.
  6. Upgrading Python version in the pre-installed environment. This usually means a large number of packages is upgraded as well.

Changes not considered as breaking

  1. Addition of new optional function argument or structure parameter.
  2. Changing a generic error to a specific one.
  3. Fixing the behavior of a function that resulted in an error.
  4. Addition of a new enumeration value.
  5. Upgrade of a Python package to its newest version in the pre-installed environment.

Exceptions

When a function call encounters an error, an exception is raised in the user code. Exceptions originating from the AutoScript server are consistently of the ApplicationServerException type. However, exceptions originating from the AutoScript client can vary in type. While ApiException is common, sometimes a more specific exception, like ValueError or IndexError, is thrown to better detail the error.

Regardless of the type, every exception includes a message explaining the error in a human-readable way.

Enumerations

Functions in the AutoScript API intentionally avoid using parameters of enumeration type (i.e., classes derived from the Enum class in Python) to describe states, opting instead for simple types like int or string. This design choice is made to ensure flexibility in both AutoScript and the underlying software platform release cycles. In scenarios where a newer version of the microscope platform introduces new states that are unknown to an older version of AutoScript, these new modes can be still provided as parameters to functions as plain strings or integers.

However, AutoScript does define enumeration classes, which provide named constants to be used in place of strings or integers. This approach helps prevent typos and enhances the readability of the code. 


        # The following lines are equivalent, the latter is preferred
        microscope.optics.lenses.get_lens("C2")
        microscope.optics.lenses.get_lens(LensType.C2)
