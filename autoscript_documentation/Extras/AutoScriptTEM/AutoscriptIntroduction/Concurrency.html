Concurrency

Support for multiple clients

AutoScript facilitates the simultaneous execution of multiple scripts. Technically speaking, the AutoScript server can handle concurrent access from multiple AutoScript clients.

The server employs an input command queue mechanism. When a scripting command is received by the server, it is placed in an input queue for processing. All connected clients share this queue, effectively serializing their requests on the microscope hardware and preventing potential collisions or unwanted interference with executed operations.

While this architecture offers clear advantages, there are also potential complications to consider. For example, a long-lasting synchronous command, such as executing a lengthy patterning job, can block the input queue for a significant period.

During this time, the AutoScript server will only queue up additional commands without processing them. If the calling script needs to maintain control over the microscope while such a lengthy action is in progress, it must opt for asynchronous control. With such approach, the script initiates the long-lasting action asynchronously, immediately freeing the input queue for other commands.

The majority of long-lasting synchronous API functions in the AutoScript API have an asynchronous alternative. You can easily identify asynchronous functions by the presence of the word "start" in their names.

Below is an example of an asynchronous command that runs stage jogging: 


        import time

        # Start jogging with X, Y axes
        microscope.specimen.piezo_stage.start_jogging(StageVelocity(x=2e-6, y=4e-6))

        while microscope.specimen.piezo_stage.is_moving:
            # Sleep for 0.25s
            time.sleep(0.25)

            # Print current velocity and position
            print(microscope.specimen.piezo_stage.jogging_velocity)
            print(microscope.specimen.piezo_stage.position)

        # The piezo stage is no longer moving if the loop ended
        print("Jogging finished")

Session resource management

The AutoScript server supports the concept of session managed resources. The resources' lifetime is bound to a specific client session. This is mainly done for resources that have to have a cleanup action performed when they are no longer needed (such as when calling stop() or when a session ends without an explicit stop call).

Currently the following resources are session managed:

  • Auto beam blank mode
  • Beam modulator

When a new client tries to modify an existing session managed resource it will take over the ownership of that resource, invalidating access from any previous client. All clients can access read only properties of the managed resource, but only one client can modify (set) it at a time.

Below is an example that demonstrates possible side effects when multiple sessions interact with a session managed resource.


        beam_modulator = microscope.optics.blanker.beam_modulator

        # 1.
        beam_modulator.start_modulation(duty_cycle=0.5, frequency=1e6)
        print(beam_modulator.duty_cycle)  # 0.5

        # 3.
        beam_modulator.stop_modulation()
        print(beam_modulator.is_modulation_on)  # !!! True, second client claimed ownership over the resource.
    

        beam_modulator = microscope.optics.blanker.beam_modulator

        # 2.
        beam_modulator.start_modulation(duty_cycle=0.2, frequency=1e6)
        print(beam_modulator.duty_cycle)  # 0.2

Support for multithreading

The AutoScript client class is designed to be thread-safe, allowing it to be accessed simultaneously from multiple threads without the need for any locking mechanism.

However, as described in the previous chapter, invoking the client object from multiple threads does not lead to parallel processing of the commands. Instead, they are still placed in the input queue on the server side and processed in series.
